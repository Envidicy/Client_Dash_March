# Envidicy Integration API v1

Read-only API для выгрузки финансовых данных и статистики клиента Envidicy.

Интерактивная документация: `https://envidicy-dash-client.onrender.com/integration/docs`

Публичная страница документации: `https://app.envidicy.kz/developers/integration-api`

Postman Collection: `https://app.envidicy.kz/envidicy-integration-api-v1.postman_collection.json`

Базовый адрес:

```text
https://envidicy-dash-client.onrender.com/api/v1/integration
```

## Возможности

API предоставляет:

- рекламные аккаунты клиента и текущие настройки комиссий;
- заявки на пополнение общего кошелька и реквизиты счетов;
- фактические движения клиентского кошелька;
- пополнения рекламных аккаунтов с комиссиями, НДС и применённым курсом;
- фактические зачисления на рекламные аккаунты и сторнирующие операции;
- текущую финансовую сводку;
- дневные расходы, показы и клики;
- текущие информационные валютные курсы.

API работает только на чтение. С помощью интеграционного ключа нельзя создавать пополнения, переводить средства, изменять баланс или настройки рекламных аккаунтов.

## Получение ключа

Управлять ключами может владелец клиентского кабинета:

1. Войти в клиентский кабинет Envidicy.
2. Открыть **Настройки → API для интеграций**.
3. Указать понятное название ключа.
4. Выбрать разрешения и срок действия.
5. Нажать **Создать API-ключ**.
6. Сразу скопировать ключ.

Полный ключ показывается только один раз. Envidicy хранит только SHA-256 хеш и не может восстановить исходный секрет. Утерянный ключ необходимо отозвать и выпустить заново.

Префикс ключей:

```text
env_live_
```

## Авторизация

Рекомендуемый заголовок:

```http
X-API-Key: env_live_your_secret_key
```

Также поддерживается Bearer:

```http
Authorization: Bearer env_live_your_secret_key
```

Пример:

```bash
curl --request GET \
  --url "https://envidicy-dash-client.onrender.com/api/v1/integration/accounts" \
  --header "Accept: application/json" \
  --header "X-API-Key: env_live_your_secret_key"
```

Не передавайте ключ в query-параметрах, браузерном JavaScript, Git-репозиториях или открытых логах.

## Разрешения

| Scope | Данные |
|---|---|
| `accounts:read` | Рекламные аккаунты, идентификаторы, статусы и текущие комиссии |
| `finance:read` | Кошелёк, пополнения, фактические зачисления, остатки и курсы |
| `performance:read` | Дневные расходы, показы и клики |

Если необходимого scope нет, API возвращает `403 Forbidden`.

## Пагинация

Списковые endpoint’ы используют cursor pagination:

| Параметр | Описание |
|---|---|
| `after_id` | Вернуть записи с ID больше указанного. По умолчанию `0` |
| `limit` | Размер страницы от `1` до `500`. По умолчанию `100` |

```json
{
  "items": [],
  "pagination": {
    "limit": 100,
    "has_more": true,
    "next_cursor": 1842
  }
}
```

Если `has_more=true`, передайте `next_cursor` как `after_id` следующего запроса.

## Endpoint’ы

### Данные интеграции

```http
GET /me
```

Возвращает ID клиента, информацию о ключе, scopes и версию API.

### Рекламные аккаунты

```http
GET /accounts
```

Scope: `accounts:read`

Ответ содержит:

- внутренний ID Envidicy;
- рекламную платформу;
- название;
- внешний ID платформы;
- код аккаунта;
- валюту и статус;
- текущую комиссию Envidicy;
- агентский ребейт;
- итоговую текущую комиссию.

```json
{
  "id": 42,
  "platform": "meta",
  "name": "Main Meta Account",
  "external_id": "act_123456",
  "account_code": "META-KZ-01",
  "currency": "USD",
  "status": "active",
  "commission": {
    "platform_fee_percent": 5.0,
    "agency_rebate_percent": 0.0,
    "total_fee_percent": 5.0,
    "source": "client_platform_default"
  }
}
```

Текущая комиссия показывает настройки на момент запроса. Для бухгалтерского учёта используйте зафиксированную комиссию конкретного пополнения из `/account-topups`.

### Заявки на пополнение кошелька

```http
GET /wallet-topups
```

Scope: `finance:read`

Фильтры:

- `status`;
- `date_from=YYYY-MM-DD`;
- `date_to=YYYY-MM-DD`.

Этот endpoint возвращает заявки и данные счетов. Наличие заявки или счёта не означает, что деньги уже зачислены. Фактическое движение необходимо брать из `/wallet-transactions`.

Основные статусы:

| Статус | Значение |
|---|---|
| `requested` | Клиент создал заявку |
| `invoice_pending` | Счёт формируется |
| `invoice_ready` | Счёт готов |
| `completed` | Средства зачислены |

### Движения кошелька

```http
GET /wallet-transactions
```

Scope: `finance:read`

Фильтры:

- `transaction_type`;
- `account_id`.

Положительная сумма увеличивает кошелёк, отрицательная уменьшает.

| Тип | Значение |
|---|---|
| `wallet_topup` | Зачисление на клиентский кошелёк |
| `topup_hold` | Резервирование средств под пополнение аккаунта |
| `topup` | Списание за завершённое пополнение аккаунта |
| `topup_hold_release` | Возврат резерва или сторно списания |
| `agency_transfer` | Перевод из кошелька агентства |
| `admin_adjustment` | Административная корректировка |

Поля `source_type` и `source_id` связывают новые проводки с исходной заявкой или пополнением. У исторических записей эти поля могут быть `null`.

### Пополнения рекламных аккаунтов

```http
GET /account-topups
```

Scope: `finance:read`

Фильтры:

- `status`;
- `account_id`;
- `date_from=YYYY-MM-DD`;
- `date_to=YYYY-MM-DD`.

Финансовые поля:

| Поле | Значение |
|---|---|
| `amounts.funding_base` | База для расчёта комиссии в валюте кошелька |
| `amounts.account_credit` | Сумма зачисления в валюте рекламного аккаунта |
| `amounts.platform_fee` | Комиссия Envidicy |
| `amounts.agency_rebate` | Агентский ребейт |
| `amounts.total_fee` | Комиссия Envidicy + агентский ребейт |
| `amounts.vat` | Сумма НДС |
| `amounts.wallet_debit` | Полное списание с кошелька |

`exchange_rate.rate` — применённый курс конкретной операции. Именно его нужно использовать для исторической операции.

### Фактические зачисления

```http
GET /funding-events
```

Scope: `finance:read`

Фильтр: `account_id`.

Это append-only реестр фактических зачислений:

- положительная сумма — зачисление;
- отрицательная — сторно;
- `source_type=topup` — зачисление по завершённому пополнению;
- `source_type=admin_manual` — ручное зачисление Envidicy;
- `reversal_for_event_id` — ID исходной операции;
- `voided_at` — исходная операция отменена.

### Финансовая сводка

```http
GET /finance/summary
```

Scope: `finance:read`

Возвращает текущий кошелёк клиента и сводку по аккаунтам: всего пополнено, расход сегодня, общий расход, расчётный остаток и доступный платформенный баланс.

Платформенный баланс может быть `null`, если рекламная система его не предоставляет.

### Дневная статистика

```http
GET /performance/daily?date_from=2026-07-01&date_to=2026-07-31
```

Scope: `performance:read`

Обе даты обязательны. Максимальный период — 366 дней. Можно передать `account_id`.

Каждая строка содержит аккаунт, дату, валюту, расход, показы, клики и время последнего обновления.

### Текущие курсы

```http
GET /exchange-rates/current
```

Scope: `finance:read`

Текущие курсы носят информационный характер. Историческую операцию нельзя пересчитывать по текущему курсу — используйте курс из `/account-topups`.

## Ошибки

```json
{
  "detail": "API key does not have required scope: finance:read"
}
```

| HTTP | Значение |
|---|---|
| `400` | Ошибка фильтров, дат или пагинации |
| `401` | Ключ отсутствует, неверен, истёк или отозван |
| `403` | Нет необходимого scope |
| `404` | Ресурс не найден |
| `429` | Превышен rate limit |
| `500` | Внутренняя ошибка |

Стандартный лимит — 600 запросов в минуту на ключ. Ответ `429` содержит `Retry-After`.

## Рекомендуемая синхронизация

1. Получить `/accounts` и сохранить внутренние ID Envidicy.
2. Читать реестры в порядке возрастания ID.
3. Сохранять `next_cursor` только после успешной обработки страницы.
4. При ошибке повторять ту же страницу.
5. Дедуплицировать данные по endpoint + `id`.
6. Фактические движения общего кошелька брать из `/wallet-transactions`.
7. Фактические зачисления на кабинеты брать из `/funding-events`.
8. Комиссии, НДС и применённый курс брать из `/account-topups`.
9. Текущие остатки получать из `/finance/summary`.

Не определяйте факт оплаты только по наличию заявки или счёта.

## Безопасность и ротация

- Создавайте отдельный ключ для каждой внешней системы.
- Выдавайте минимально необходимые scopes.
- Храните ключ в серверном secret manager.
- Выпускайте новый ключ до истечения старого.
- Сначала переключайте интеграцию на новый ключ, затем отзывайте старый.
- Отзыв применяется немедленно.
- Envidicy записывает в аудит ключ, путь, HTTP-метод, статус ответа и IP-адрес.

## Версионирование

Стабильная версия — `v1`. Несовместимые изменения будут выпускаться под новым URL. В ответы могут добавляться новые поля без изменения версии, поэтому потребитель должен игнорировать неизвестные поля.
